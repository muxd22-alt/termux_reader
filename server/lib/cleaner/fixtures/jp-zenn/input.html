Dockerマルチステージビルドの実践ガイド

dev_user
2024-03-10 10分で読める

Dockerのマルチステージビルドは、最終的なイメージサイズを大幅に削減するための重要なテクニックです。この記事では、実際のプロジェクトでマルチステージビルドを活用する方法を解説します。

なぜマルチステージビルドが必要なのか

シングルステージビルドでは、ビルドツールやコンパイラがイメージに含まれてしまい、最終イメージが不必要に大きくなります。マルチステージビルドでは、ビルド用のステージと実行用のステージを分離することで、この問題を解決します。

# Build stage
FROM golang:1.22 AS builder
WORKDIR /app
COPY . .
RUN go build -o server .

# Runtime stage
FROM alpine:3.19
COPY --from=builder /app/server /usr/local/bin/
CMD ["server"]

Go アプリケーションの例

Goアプリケーションは静的リンクされたバイナリを生成できるため、マルチステージビルドとの相性が非常に良いです。ビルドステージでコンパイルし、実行ステージでは軽量なalpineイメージにバイナリだけをコピーします。

Node.js アプリケーションの例

Node.jsの場合は、ビルドステージでnpm ciとビルドを実行し、実行ステージではproduction dependenciesのみをインストールします。これにより、devDependenciesやビルドキャッシュを除外できます。

ベストプラクティス

.dockerignoreファイルを適切に設定し、node_modulesやテストファイルをビルドコンテキストから除外することも重要です。また、キャッシュを活用するためにCOPYの順序にも注意が必要です。
